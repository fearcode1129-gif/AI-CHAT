## 数据库设计

**1. Note（笔记表）**

- id Int @id @default(autoincrement())：主键，自增
- title String @default("")：标题
- content String @default("")：正文（HTML）
- isFavorite Boolean @default(false)：是否收藏
- isTrashed Boolean @default(false)：是否在回收站
- trashedAt DateTime?：移入回收站时间（可空）
- createdAt DateTime @default(now())：创建时间
- updatedAt DateTime @updatedAt：更新时间（自动维护）
- tags Tag[] @relation("NotesTags")：关联多个标签

**2. Tag（标签表）**

- id Int @id @default(autoincrement())：主键
- name String @unique：标签名（唯一）
- createdAt DateTime @default(now())：创建时间
- updatedAt DateTime @updatedAt：更新时间
- notes Note[] @relation("NotesTags")：关联多篇笔记

**3. Note <-> Tag 关系**

- 是多对多（一个笔记多个标签，一个标签多篇笔记）

# **简历项目问题**

## 1.**AI流式输出如何实现?**

SSE 是一种基于 HTTP 的服务端推送技术。客户端发起请求后，服务端返回 `text/event-stream`，并保持连接不断开；之后服务端每产生一段新数据，就按 `data: xxx\n\n` 的格式持续写回并 flush，客户端就能实时接收，从而实现流式输出。它本质上是利用 HTTP 长连接和分块传输实现边生产边消费。SSE 适合服务端单向推送场景，比如消息通知、日志流和大模型流式回答，**SSE 的断线重连**分两层。第一层是**连接恢复**，浏览器原生 `EventSource` 断开后会自动重连，服务端也可以通过 `retry` 指定**重连间隔**。第二层是数据续传，服务端发送消息时要带 `id`，客户端断线重连时会自动携带 `Last-Event-ID`，服务端根据这个 ID 从断点后继续推送，避免消息丢失。

#### **SSE优化:**

**连接层**：重复建连、心跳、断线重连、代理层缓冲

**数据层**：token 颗粒度太细，前端解析和状态更新过于频繁

**渲染层**：每来一个 chunk 就 setState，导致高频 re-render

**架构层**：消息列表很大、Markdown 解析重、代码高亮重、状态管理扩散

#### 1.chunk -> buffer -> 节流 flush(最后一次消息频繁更新)

在流式输出场景里，后端会连续返回很多很小的 chunk。如果前端每收到一个 chunk 就立刻更新一次状态和界面，会导致非常高频的重渲染，主线程压力很大，容易出现卡顿、掉帧、输入不跟手等问题。`chunk -> buffer -> 节流 flush` 的思路是，**先把收到的 chunk 暂存到 buffer 里**，不直接渲染，再通过**节流**机制按固定时间间隔把 buffer 中累计的数据合并后一次性刷新到 UI。这样就把原本很多次零碎更新，变成了少量批量更新。它的核心价值在于降低状态更新次数、减少组件重渲染和布局计算，同时又能保留流式输出的观感，所以在 SSE、WebSocket、AI 对话这类前端场景里非常常见。

#### 2.消息列表虚拟化(长会话)

消息列表虚拟化主要是解决**长列表场景下的渲染性能问题**。比如聊天页面里，如果有几百条甚至上千条消息，前端如果把所有消息一次性都渲染到 DOM 中，会带来很高的内存占用、组件渲染开销和浏览器布局计算成本，尤其是在消息持续增加或者最后一条消息还在流式更新时，页面会越来越卡。**虚拟化的核心思路是只渲染当前可视区域附近的消息**，屏幕外的内容不真正挂载到 DOM，而是通过占位高度来维持整体滚动效果。这样可以显著减少页面中的真实节点数量，**降低 diff、reflow 和 repaint 的压力**。对于聊天场景来说，它的价值主要体现在会话很长时依然能保持滚动流畅、内存稳定，以及避免随着消息量增长性能持续下降。

#### 3.状态管理不要让 SSE 更新扩散到全局

**SSE 属于高频更新数据**，如果把它放到全局状态里，**每来一个 chunk 都可能触发很多无关组件一起更新**，造成不必要的**渲染开销**。更合理的做法是把**流式消息状态收敛在聊天模块内部**，最好只让当前正在生成的那条消息订阅这部分数据，历史消息和页面其他区域不要被动更新。这样可以把 SSE 的影响范围控制在最小，避免高频状态变更扩散到整棵组件树。

## 2.实现语义搜索能力，如何实现?

我现在的检索增强方案，本质上是“**关键词检索 + 语义召回**”的混合搜索。实现时，**先把每篇笔记的标题、标签和正文统一转成可检索文本**，其中正文会先去掉富文本标签，再做标准化处理。随后**按段落优先**、**固定长度补充**的方式把长文本切成多个 **chunk**，避免整篇笔记过长导致语义被稀释。用户搜索时，前端会把查询词和当前笔记数据发送到后端，后端调用 embedding 模型，将 **query 和所有 chunk** 一起向量化，再计算 query 与各个 chunk 的相似度，并为每篇笔记保留得分最高的那个 chunk 作为语义命中结果和摘要片段。最终排序时，并不是只用向量结果，而是把标题命中、正文命中、标签命中的关键词分数，与语义相似度分数融合，得到综合排序结果。这样既能保证术语、标题类搜索的精确性，也能支持自然语言表达下的模糊召回。

#### 用的什么向量数据库?

这个项目继续使用 SQLite，我会选择 **sqlite-vec** 作为向量检索方案。因为它是专门给 SQLite 增加向量检索能力的扩展，比较适合我这种本地优先、单机使用的 Notes 产品。这样我就不用为了语义检索额外引入一套独立向量数据库，主数据和向量检索都可以继续围绕 SQLite 来组织。如果以后项目复杂的话我会选择Milvus

#### 优化：

RAG 检索，核心思路是把“查询时现算”改成“写入时建索引，查询时直接召回”。具体来说，**我会先把笔记按 chunk 拆分，并把每个 chunk 的文本、摘要和 embedding 持久化到数据库**，而不是每次搜索时再重新切块和向量化全部笔记。查询阶段只需要对用户输入的 **query 做一次 embedding**，然后直接在已有的 chunk 向量上做相似度召回，这样可以明显降低搜索延迟和外部 embedding 调用成本。

为了避免每次保存笔记都全量重建索引，我会给笔记内容或者 **chunk 内容加一个 contentHash**。它本质上是内容的哈希指纹，用来判断文本是否真的发生变化。如果 hash 没变，就直接复用原有 chunk 和 embedding；只有 hash 变化时，才重新切块和更新向量，这样可以把**全量更新优化成增量更新**。

## **3.采用本地编辑状态 + 防抖自动保存方案，同时修复了多窗口同步、跳转联动、长文滚动下工具栏定位问题如何实现**

自动保存要做防抖，是因为编辑输入是高频操作，如果每次输入都请求后端，请求量会很大，还可能出现旧请求覆盖新内容的问题。
 所以我的做法是把状态分成两层：一层是本地实时编辑状态，保证输入流畅；另一层是服务端已保存状态。用户输入时先只更新本地状态并标记 dirty，等用户停止输入一段时间后再通过防抖触发自动保存。
 多窗口同步我会用 `storage` 事件或者 `BroadcastChannel` 通知其他标签页当前笔记已经更新，如果对方没有本地未保存修改就直接同步，否则给冲突提示。
 为了避免内容丢失，自动保存失败时我不会清空本地草稿，还会把草稿按 `noteId` 持久化到 localStorage 或 IndexedDB，并配合重试和错误提示，保证用户内容能恢复。

## **4.如何防护xss?**

这三个问题本质上都是 XSS 风险。因为笔记内容最终会被再次渲染，如果直接信任用户输入，就可能执行恶意脚本。
 我的处理原则是输入可以保留，但输出必须净化。我不会直接把用户原始内容插入 DOM，而是会在渲染前做白名单过滤和 sanitize，只保留安全标签和安全属性，过滤掉 `script`、内联事件和危险协议。
 对 Markdown，我会先解析再净化，不会把转换出来的 HTML 直接渲染；对富文本，我会对生成的 HTML 做统一的安全过滤，避免脚本注入。
 如果必须使用 `dangerouslySetInnerHTML`，也一定是在内容已经经过严格净化之后。
 同时后端也应该有一层兜底校验，因为真正的安全边界不能只依赖前端。

## 5.大纲章节跳转

本质上是“**编辑器文档结构、正文滚动位置、侧边栏状态**”三者联动。

具体做法是：

1. 先从 TipTap/ProseMirror 文档里遍历所有标题节点，提取出
   - 标题文本
   - 标题层级
   - 标题在文档里的 pos 位置
     这样生成一份结构化的大纲数据。
2. 然后把这份大纲数据同步给左侧侧边栏，同时给正文里的真实标题 DOM 挂上 data-outline-pos 这样的定位标记。
3. 正文滚动时，我监听右侧滚动容器，根据当前视口顶部附近“最后一个经过阈值线的标题”，判断当前用户正在阅读哪一节，并把它对应的 pos 写回全局状态。
4. 左侧侧边栏根据这个 activeOutlinePos 去做高亮渲染；当高亮项变化时，再用 scrollIntoView 把当前目录项自动滚到可视区域。
5. 反过来，点击左侧目录时，我会直接调用编辑器的 setTextSelection(pos + 1).scrollIntoView()，让正文跳到对应标题位置。
